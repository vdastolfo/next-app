1 Visión General del Sistema

Next Subastas es una plataforma de subastas online donde los usuarios pueden registrarse, asistir a subastas en tiempo real y pujar por lotes. El sistema se divide en dos aplicaciones independientes que se comunican a través de una API REST y un canal WebSocket.

Arquitectura General

📱 Frontend (Expo)  ↔  🔌 API REST + WebSocket  ↔  ⚙️ Backend (Spring Boot)  ↔  🗄️ SQL Server
Frontend corre en el dispositivo móvil. Realiza peticiones HTTP al backend usando Axios. Se suscribe a tópicos WebSocket para recibir actualizaciones de pujas en tiempo real.

Backend expone endpoints REST bajo /api/** y un broker STOMP en /ws. Procesa la lógica de negocio, persiste en SQL Server y corre un scheduler que avanza los ítems automáticamente.

Base de datos SQL Server Express en el puerto 1433, base NextSubastas. Hibernate con ddl-auto=update gestiona el schema automáticamente.

Capa: config/

🔒 SecurityConfig.java

Define qué endpoints son públicos y cuáles requieren JWT. Registra el filtro JwtAuthenticationFilter que intercepta cada request, extrae el token del header Authorization: Bearer ... y carga el usuario en el contexto de Spring Security. Las rutas /api/auth/** y /fotos/** son públicas; todo lo demás requiere autenticación.

🔌 WebSocketConfig.java

Configura un broker STOMP en el endpoint /ws. Los clientes se suscriben a tópicos como /topic/item/{itemId} para recibir actualizaciones de pujas en tiempo real, y /topic/subasta/{subastaId} para saber cuándo cambia el ítem activo.

⏱ ItemActivoScheduler.java

Se ejecuta cada 10 segundos. Para cada subasta con estado abierta cuya fecha/hora ya pasó: si no tiene ítem activo, activa el primero disponible. Si el ítem activo superó su fechaFin, llama a cerrarItemActivo() que determina el ganador, envía el email y avanza al siguiente ítem.

🖼 FotoDataLoader.java

Al iniciar el backend, escanea resources/fotos/{productoId}/*.jpg. Si el producto existe en DB y sus fotos actuales son placeholders (≤100 bytes), las reemplaza con los archivos reales. Las carpetas deben llamarse con el ID numérico del producto.

3 Modelo de Datos (Entidades)

Todas las entidades se mapean a tablas en SQL Server. Hibernate usa PhysicalNamingStrategyStandardImpl, por lo que los nombres de columna coinciden exactamente con los nombres de campo en Java (camelCase).

Jerarquía central de Persona

Persona es la entidad raíz. UsuarioApp, Cliente, Subastador y Duenio son "extensiones" de Persona usando @MapsId — comparten el mismo ID que Persona.

AuthContext.js

Estado global de autenticación

Provee user, login(token) y logout() a toda la app mediante React Context. El JWT se guarda en AsyncStorage. Al iniciar la app, intenta restaurar la sesión desde storage. El interceptor de Axios lee el token de AsyncStorage en cada request automáticamente.

7 Reglas de Negocio Clave

Moneda de subastas

💱 Subastas bimonetarias prohibidas

Cada subasta tiene una moneda fija (pesos o dólares). Las subastas en dólares solo admiten pago con tarjeta de crédito o cuenta bancaria — el cheque está bloqueado. Esta validación ocurre en SubastaService.pagarPujaGanada().

Ítems sin ofertas

🏢 compradoPorEmpresa

Cuando un ítem llega al final del timer sin ninguna puja, se marca subastado='si' y compradoPorEmpresa=true — indica que la empresa lo adquiere a precio base. No se envía email de ganador.

Subida de categoría (automática)

📈 Evaluación al consultar participaciones

Cada vez que un usuario accede a la pantalla de participaciones, el backend evalúa si puede subir de categoría. Condiciones: (1) ≥ 3 lotes ganados en la categoría actual y (2) ≥ 3 medios de pago verificados. Si se cumplen ambas, sube automáticamente y la respuesta incluye subioCategoria: true.

Fotos de perfil

👤 Base64 en NVARCHAR(MAX)

La foto de perfil se almacena como cadena base64 en el campo Persona.fotoPerfil (columna NVARCHAR(MAX)). Se sube con timeout extendido de 30 segundos por el tamaño. El frontend usa expo-image-picker con calidad 0.5 y crop 1:1.
